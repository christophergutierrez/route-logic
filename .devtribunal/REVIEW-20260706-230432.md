# Code Review Tribunal — route-logic (pass 1)

**Date:** 2026-07-06
**Scope:** `wip` — all untracked source in `route-logic/` (the pass-1 data-engine runner)
**Mode:** review-only (no converge)
**Reviewers:** Language/correctness, Security/config, Tests/gate-behavior, Architect synthesis
**Verdict:** ⚠️ **FAIL (blocking)** — 1 critical, 2 high. The runner works end-to-end in mock

## Tests / tools run

- `python3 -m py_compile bin/*.py tasks/add_two/*.py` → **OK** (all compile)
- `python3 bin/run_bracket.py --record tasks/add_two/record.json --mock` → **OK**, prints
  `fast=FAIL(1) standard=PASS(0) reasoning=PASS(0)` → `minimum viable tier: standard`, exit 0
- golden/buggy gate discrimination: `golden.py` → test exits 0; `buggy.py` → test exits 1

- `classify.minimum_viable_tier()` edge cases: all-fail→`NONE`, all-pass→`fast`,

- `classify.minimum_viable_tier()` fed run_bracket's bracket shape (`{tier, verdict}`) →
  **silently returns `NONE`** (schema mismatch — see C1).

## Findings

### C1 — CRITICAL — run_bracket and classify use incompatible bracket schemas (silently wrong label)

**File:** `bin/run_bracket.py:85,148-149` vs `bin/classify.py:78`
**Category:** contract / correctness

`classify.py`'s docstring (line 11) states it turns *"`bin/run_bracket.py`'s bracket* — one
pass/fail result per tier — into the single label." But the two never connect and disagree on
shape:

- `run_bracket.py` result objects carry `"verdict": "PASS"|"FAIL"` (line 85) and compute the
  label inline: `passing = {r["tier"] for r in results if r["verdict"] == "PASS"}` (line 148).
- `classify.py`'s `_normalize` reads `entry.get("passed", False)` (line 78) — a **boolean**,
  not a verdict string. It accepts `{tier: bool}` dicts or `[{tier, passed}]` lists.

Verified: feeding run_bracket's actual bracket shape into `classify.minimum_viable_tier()`
returns `NONE` for a bracket where standard and reasoning passed — **silently wrong**. The
`passed` key is absent, so `_is_passed(entry.get("passed", False))` → `_is_passed(False)` →
`False` for every tier.

`run_bracket.py` also never writes a bracket file, so `classify.py` (a file/stdin CLI) has no
producer. The project's core output — "minimum viable tier" — is computed by two independent,
drifting code paths with no shared contract.

**Fix:** pick one bracket schema and make run_bracket the producer. Minimal: have run_bracket
emit a bracket JSON to stdout (or `--out`) using the `passed`-bool shape classify expects, and
have run_bracket call `classify.minimum_viable_tier()` instead of its inline reimplementation.
Delete the inline logic (line 148-149) so there is one labeling path.

### H1 — HIGH — gate subprocess has no timeout (model-generated code can hang the bracket forever)

**File:** `bin/run_bracket.py:78-82`
**Category:** resource / correctness

```python
proc = subprocess.run(
    gate["command"], shell=True,
    cwd=str(gr._resolve_cwd(sandbox, gate["cwd"])),
    capture_output=True, text=True,
)   # no timeout=
```

The gate runs model-generated Python (`python3 test_src.py`). A model that emits `while True:
pass` or `import time; time.sleep(10**9)` hangs the bracket indefinitely. The live model call in
`executor.py:65` correctly has `timeout=180`, but the gate — which executes untrusted output —
has none. For a data engine meant to run many tasks unattended, this is a hang/livelock risk.

**Fix:** add `timeout=` to the gate `subprocess.run` (e.g. 60s for pass 1) and catch
`subprocess.TimeoutExpired` → verdict `"FAIL"` (or `"TIMEOUT"`).

### H2 — HIGH — run_bracket depends on killhouse private API (`gr._materialize_pinned_artifacts`, `gr._resolve_cwd`)

**File:** `bin/run_bracket.py:72,80`
**Category:** contract / maintainability

```python
gr._materialize_pinned_artifacts(record, sandbox)   # line 72 — private
cwd=str(gr._resolve_cwd(sandbox, gate["cwd"])),      # line 80 — private
```

These are underscore-prefixed internals of `killhouse_gate_replay`. The README/CONTEXT claim
route-logic "does NOT reimplement killhouse; it reuses" it — but reaching into private symbols
is the opposite of reuse: any killhouse refactor of those helpers silently breaks this runner
with an `AttributeError` at runtime, not at import. There's no version pin or contract test
guarding this.

**Fix:** either (a) ask killhouse to expose public sandbox helpers (`materialize_pinned_artifacts`,
`resolve_cwd`) and call those, or (b) vendor the ~15 lines of logic locally with a comment
naming the killhouse version they mirror. Prefer (a).

### M1 — MEDIUM — `os.environ["RE_TIER"]` mutated globally across tiers (shared process state)

**File:** `bin/run_bracket.py:67`
**Category:** correctness / concurrency

```python
os.environ["RE_TIER"] = tier  # the mock executor keys off this; harmless in live mode
```

`run_tier` mutates the process environment on every call. Tiers run sequentially today so it
works, but the comment "harmless in live mode" papers over a latent bug: if tiers are ever run
in parallel (an obvious optimization for a data engine), `RE_TIER` becomes a race and the mock
executor reads the wrong tier's value. Env mutation as inter-call parameter passing is fragile.

**Fix:** pass tier through the executor template/env of the subprocess rather than the parent
process env, or document that tiers must run sequentially.

---

### M2 — MEDIUM — ERROR/SKIPPED tiers are silently labeled as FAIL (conflates "unmeasured" with "failed")

**File:** `bin/run_bracket.py:148`
**Category:** semantics

`passing = {r["tier"] for r in results if r["verdict"] == "PASS"}` treats every non-PASS
verdict — `FAIL`, `ERROR` (executor/model failure), `SKIPPED_NO_ROUTING` — identically. A task
where the model call errored on all tiers gets labeled `NONE` ("no tier passed"), but the
correct label is "unmeasured." For a dataset whose entire value is accurate pass/fail-from-a-
real-gate labels, recording an infra failure as a gate failure corrupts the corpus. The
run_bracket.py:131 comment even acknowledges this class of risk for empty base_url.

**Fix:** distinguish `ERROR`/`SKIPPED` from `FAIL` in the bracket output and have classify
surface "UNMEASURED" rather than collapsing to `NONE`.

---

### M3 — MEDIUM — no tests for the orchestrator/labeler (classify.py is pure and trivially testable)

**File:** `bin/classify.py` (whole), `bin/run_bracket.py` (whole)
**Category:** coverage

The only test in the repo is the task fixture (`tasks/add_two/test_src.py`). `classify.py` is
explicitly documented as "pure … so it can be unit-tested and replayed freely" (line 15) but has
zero tests — and C1 shows its schema contract is in fact broken, which a single unit test
feeding run_bracket's shape would have caught. The mock bracket run is a de facto integration
test but asserts nothing programmatically (a human reads stdout).

**Fix:** add `tests/test_classify.py` covering the dict/list shapes, all-fail/all-pass/edge
cases, and — critically — the run_bracket bracket shape. Add a mock-bracket smoke test that
asserts exit 0 and `standard` in stdout.

### L1 — LOW — `extract_code` returns whole response when unfenced (executor writes arbitrary prose as code)

**File:** `bin/executor.py:42-45`
**Category:** robustness

```python
def extract_code(text: str) -> str:
    m = _FENCE.search(text)
    return m.group(1) if m else text   # unfenced → write the entire response as src.py
```

If a model returns prose with no fence, the entire response (including chatty preamble) is
written to `src.py` and then executed by the gate. For pass 1 this is "harden later" per the
docstring, but it means a non-fencing model produces a `SyntaxError` gate FAIL that looks like a
capability failure — a labeling artifact.

**Fix:** when unfenced, either fail the executor (non-zero exit → `ERROR`) or attempt to extract
the first `def ` block. At minimum log that no fence was found.

## Architect synthesis

**[High-Level Summary]**
The runner is well-structured for a pass-1 toy: stdlib-only, clean separation of runner /
executor / labeler, a working mock mode, and a gate that genuinely discriminates. The highest-
risk area is **not** the gate or the model call — it's the **labeling contract**. The project
exists to produce one label ("minimum viable tier"), and that label is computed by two code
paths that don't share a schema and don't connect. C1 is a silent-correctness bug: the module
documented as the labeler returns `NONE` for a passing bracket.

**[Cross-Cutting Concerns]**
* **Theme:** two implementations of the same core concept (minimum viable tier) with no shared
  contract.
* **Type:** risk
* **Severity:** high
* **Confidence:** confirmed
* **Related Findings:** C1, M3
* **Observation:** `run_bracket.py` computes the label inline (line 148-149) using a `verdict`
  string; `classify.py` computes it from a `passed` bool. The docstrings claim one feeds the
  other, but they don't. Both the schema drift (C1) and the missing tests (M3) are symptoms of
  the same root cause: there is no canonical bracket type.
* **Recommendation:** define one bracket schema (suggest `[{tier, passed, exit, model}]`),
  make `run_bracket.py` emit it, make `classify.py` the sole labeler, delete the inline logic,
  and add the test that would have caught C1.

**[Severity Overrides]**
None. The Language reviewer's findings and the Security reviewer's findings (gate timeout,
untrusted-output execution) are all confirmed and kept at their severities. No escalations or
dismissals.

**[Ponytail check]**
The diff is appropriately minimal for pass 1 — no speculative abstractions, no pass-through
wrappers, no new dependencies (stdlib only). `classify.py` is slightly over-built for current
use (two input shapes, strict-truthiness docstring) but that's defensible since it's the
labeled-corpus contract. Nothing to delete.
